# BUG-129 — Intake Document (Gate 0 + Gate 1)

**Status:** REGISTERED · **Priority:** P1 · **Sprint:** POS 4.0 · **Date:** 2026-06-11
**Source:** Insights audits — cafe103 (A2) + Palm House (A3)
**Type:** Bug — **BACKEND ASK** (no frontend code)

---

## 1. Symptom
TAB (credit) orders are stamped `f_order_status = 6` ("paid") at punch time, before any money is collected. Every frontend gate keyed on fStatus 6 (Sales, Payments, Dashboard, Items & Menu) therefore counts uncollected credit as settled revenue.

## 2. Verified Impact
| Restaurant | TAB orders Mar–Jun | Value shown as revenue |
|---|---|---|
| Palm House | 319 (all fStatus=6) | ₹1,71,023 |
| cafe103 | 141 (all fStatus=6) | ₹5,75,000+ (Mar ₹3,44,698 + May ₹2,30,939 + …) |

Backend's own `daily-sales-revenue-report` treats TAB as `unpaid_revenue`/`orderTAB` until `tab_cash/card/upi` settlement — i.e., the backend report engine and the backend status stamp **contradict each other**.

## 3. Ask (backend team)
1. Why is fStatus 6 assigned to TAB at punch? Is a distinct status (or reliable `payment_status`) available on the list payload?
2. Frontend needs a way to distinguish settled vs outstanding credit from `/order-logs-report` without joining the settlement endpoint.

## 4. Scope
- Area: Backend — order lifecycle / status stamping
- Blocks: CR-030 (attribution), BUG-127 (Unsettled TAB tile), TAB gate alignment in Sales/Payments/Dashboard
