# BUG-128 — Intake Document (Gate 0 + Gate 1)

**Status:** REGISTERED · **Priority:** P2 · **Sprint:** POS 4.0 · **Date:** 2026-06-11
**Source:** Insights audits — cafe103 (A7) + Palm House (A8)
**Type:** Bug (performance + latent correctness)

---

## 1. Symptom
Dashboard load is ~2× slower/heavier than necessary: it POSTs the identical `/order-logs-report` request twice per load. A Palm House month is ~40 MB → ~80 MB transferred for one dashboard view.

## 2. Root Cause (verified)
`insightsService.js` getDashboardAggregated (~lines 530-541): `ordersResp` and `cancelDataResp` are **byte-identical** POSTs (`sort_by: 'created_at'`, same `from_date`/`to_date`). The comment says the second call is for "cancel_at attribution (Option A: separate API call)", but no wider window is fetched.

## 3. Latent Correctness Issue
The cancel-attribution intent (fetch a wider window so cancellations of orders *created before the range* are caught) is NOT implemented — the Dashboard misses cancels of pre-range orders. Live-verified gap (proven settlement lags of 3+ weeks at Palm House — order 014894 punched May 16, actioned June).

## 4. Scope
- Area: Reports / Insights — Dashboard
- File: `frontend/src/api/services/insightsService.js`
- Fix direction: drop the duplicate call (reuse one response) — small; widening the cancel window is a follow-up decision (interacts with CR-030/CR-031).
