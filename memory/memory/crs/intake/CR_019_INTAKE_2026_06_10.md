## 1. Problem Statement (Owner Verbatim)

> I want to register one CR, which will be basically the restaurant settings. So this will be the first page where the user will come and set up his restaurant. So this will be a new module, new CR. It should be like a wizard. Step number one, step number two, step number three, that kind of view. The most important fields which are required should be on step number one. Design should be very easy, user-friendly because restaurant will be self-on-boarding it.

## 2. Summary

| Aspect | Detail |
|---|---|
| **Current state** | Settings exist as a slide-over panel from Dashboard (`SettingsPanel.jsx`) with 12 tiles. No standalone setup page. No onboarding flow. Restaurant info is view/edit of existing data — not a first-time wizard. |
| **Target** | New standalone wizard-style page (route: `/restaurant-settings`) for restaurant self-onboarding. 6-step wizard with mandatory/optional step distinction. Pre-populates from API, saves via `update-settings`. |
| **Backend** | APIs already exist: `GET settings-list` + `POST update-settings`. No backend changes needed. |

## 3. API Endpoints

### GET — Fetch Current Settings
```
GET /api/v2/vendoremployee/restaurant-settings/settings-list
Headers: Authorization: Bearer {token}
Response: { success: true, data: { basic: {...}, advanced: {...}, vendor: {...} } }
```

### POST — Update Settings (multipart/form-data)
```
POST /api/v2/vendoremployee/restaurant-settings/update-settings
Headers: Authorization: Bearer {token}
Body (multipart):
  - data: JSON string with { basic, advanced, vendor } sections
  - logo: file (optional) — restaurant logo image
  - pdf: file (optional) — PDF menu
```

### API Response Shape (GET)
```json
{
```

## 4. Locked Decisions (from brainstorm session)

| # | Question | Decision |
|---|---|---|
| 1 | Page type | **Standalone route** `/restaurant-settings` (not slide-over) |
| 2 | Wizard steps | **6 steps** (see §5 below) |
| 3 | Required vs Optional | **Steps 1, 2, 6 = required** (no skip). **Steps 3, 4, 5 = optional** ("Skip for now" applies defaults). |
| 4 | Data source | **Pre-populate from GET `settings-list`** — edit existing, not create-from-scratch |
| 5 | Save behavior | **Each step saves on "Save & Continue"** via POST `update-settings` |
| 6 | Navigation | **Left-rail vertical stepper** (green ✓ completed, orange active, gray upcoming) |
| 7 | UX priority | **Dead-simple, self-service** — restaurant owners onboard without help |
| 8 | Sidebar entry | **Under Settings** in sidebar |
| 9 | Access | **Same auth as existing Settings** — no new permission gating |
| 10 | Relationship to SettingsPanel | **Separate page** — existing SettingsPanel (slide-over) remains for day-to-day changes |

## 6. UX Design Decisions

| Decision | Choice |
|---|---|
| Layout | Left-rail stepper (280px) + main content area |
| Step indicators | Green ✓ (completed), Orange circle (active), Gray circle (upcoming) |
| Required/Optional markers | Orange banner on required steps, Green banner on optional steps |
| Skip button | "Skip for now →" on Steps 3, 4, 5 only |
| Field completion feedback | Green tint background on filled fields |
| Service channels | Visual card selection (4 cards: Dine-In, Takeaway, Delivery, Room) |
| Payment methods | Chip-style selectors with checkmarks |
| Conditional fields | Appear/hide based on parent toggle (e.g., GST fields when GST ON) |
| Final step CTA | "Save & Launch" (green button) |
| Validation | Per-step validation before proceeding; mandatory fields block "Save & Continue" |

## 8. Scope

### In Scope
- 6-step wizard page at `/restaurant-settings`
- Pre-populate from `GET settings-list`
- Save via `POST update-settings` (multipart/form-data)
- Logo + PDF file uploads
- Left-rail progress stepper
- Skip for optional steps (3, 4, 5)
- Per-step validation (mandatory fields)
- Sidebar entry
- Route protection (ProtectedRoute)

### Out of Scope
- New backend endpoints (using existing APIs)
- Changes to existing SettingsPanel (slide-over remains for day-to-day)
- Auto-redirect for first-time users (future enhancement)
- Operating hours / schedules management (separate API, not in settings-list)
- Table management (separate API)
- Printer management (separate API)
- Cancellation reasons management (separate API)
