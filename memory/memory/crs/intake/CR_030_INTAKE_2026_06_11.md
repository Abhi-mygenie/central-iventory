# CR-030 — Intake Document (Gate 0 + Gate 1)

**Status:** REGISTERED · **Priority:** P1 · **Sprint:** POS 4.0 · **Date:** 2026-06-11
**Source:** Insights audits — cafe103 (A1, #1 complaint source) + Palm House (A2)
**Type:** CR — requires OWNER DECISION before Gate 3

---

## 1. Requirement
One canonical revenue attribution (punch date vs collection date) — or a uniform, labelled toggle on every Insights screen. Today the module ships both silently.

## 2. Current State (verified)
- **Punch date (`created_at`):** Sales, Payments, Dashboard, Items & Menu, Order Ledger, Cancellations, Kitchen Ops, Food Court, Room Orders
- **Collection date (`collect_bill`):** Settlement Report, backend Order Summary (`daily-sales-revenue-report`)

Palm House live proof:
| Day | Punch (Sales) | Collection (backend / settlement) |
|---|---|---|
| Jun 2 | 37,422 | 49,180 / 51,494 |
| Jun 3 | 7,838 | 22,897 / 24,892 |
| Jun 7 | 0 | 2,705 / 2,631 |
| Mar 15 | 51,064 | 64,648 / 68,420 |

Extreme case: order `014894` punched **May 16**, collected **Jun 8** — Sales books it in May (closed month), settlement in June.

## 3. Owner Decision Needed (blocks Gate 3)
OD-1: Is "revenue" billed (punch) or collected (settlement) for this business?
OD-2: One canonical basis everywhere, or labelled toggle per screen?
OD-3: TAB recognised at punch or settlement? (interacts with BUG-129)

## 4. Scope
- All Insights screens + `orderLedgerService.sortBy` plumbing (toggle already half-exists: `collect_bill | created_at | cancel_at` in insightsService)
- Label component: "by punch date / by collection date" badge
