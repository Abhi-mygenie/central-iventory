## 3. Phase 2A API Mapping — Frozen

### 3A. Statement Preview + PDF — Data Sources

| Data | Source | API | Available? |
|---|---|---|---|
| Customer name, mobile | SS1 selection → `customer.name`, `customer.mobile` | API 1 | YES |
| Outstanding balance | SS1 selection → `customer.balance` | API 1 | YES |
| All credit entries | API 2 → `customer-transaction-list` | `GET tap-customer-record-list?customer_id={id}` | YES |
| All debit entries | API 2 → `customer-transaction-list-debit` | Same | YES |
| Summary dates | API 2 → `tap_start_date`, `last_tap_credit_date/amount`, `last_tap_debit_date/amount` | Same | YES |
| Total credit | Derived: `sum(credit_order_amount)` | From API 2 credits | YES (derived) |
| Total paid | Derived: `sum(debit_order_amount)` | From API 2 debits | YES (derived) |
| Bill line items (Q3=C) | `getSingleOrderNew(order_id)` per credit entry | `POST get-single-order-new` | YES (serial N calls) |
| Restaurant name | `RestaurantContext → restaurant.name` | Already loaded | YES |
| Cashier/generated by | `AuthContext → user` | Already loaded | YES |

### 3B. Date Range Filter — Client-side Implementation

| Aspect | Value |
|---|---|
| Filter targets | `created_at` / `order_created_at` on credits, `created_at` on debits |
| Input | Two `<input type="date">` (from/to) |
| Behavior | Filter transactions list client-side. Summary tiles show filtered totals. |
| Opening balance | NOT computed in Phase 2A (would require BG-02 — Phase 2B) |
| Clear | "Clear" button resets both dates |

| Outstanding | `sum(balance)` from customer list | **YES** (always available) |
