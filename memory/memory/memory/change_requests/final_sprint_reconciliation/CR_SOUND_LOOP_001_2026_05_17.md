# Change Request: Looping Notification Sound Until Order Action

**CR ID:** CR-SOUND-LOOP-001
**Date:** 2026-05-17
**Requested by:** Owner
**Priority:** P1
**Sprint:** Next phase (post POS2.0 Wave 6)

---

## 1. Current Behavior

- FCM notification arrives with a sound key (e.g., `confirm_order`, `five_sec_buzzer`, `forty_five_sec_buzzer`)
- `NotificationContext` resolves the sound key and calls `soundManager.play(soundKey)`
- The WAV file plays **once**, for its full duration (5s / 10s / 45s depending on file)
- Sound ends naturally when the file finishes
- No looping — cashier may miss the notification if they step away

## 2. Desired Behavior

- Use a **single short sound clip (~3 seconds)**
- **Loop it continuously** until the cashier takes action on the order
- Sound stops ONLY when:
  - Order is **accepted** (confirm)
  - Order is **rejected** (cancel)
  - Order is **snoozed/muted** (snooze button)
  - A **silent notification** arrives
  - Cashier **logs out** or page refreshes

## 3. Scope

### Files to change

| File | Change |
|---|---|
| `frontend/src/utils/soundManager.js` | Add `playLoop(soundKey)` method using HTML5 `audio.loop = true` |
| `frontend/src/contexts/NotificationContext.jsx` | For confirm-order/YTC notifications, call `soundManager.playLoop()` instead of `soundManager.play()` |
| `frontend/src/components/dashboard/ScanOrderPopOut.jsx` | Ensure `soundManager.stop()` is called on Accept and Reject clicks (Snooze already calls it) |

### Files NOT changed

| File | Reason |
|---|---|
| Sound files on disk | Reuse existing `confirm_order.wav` (~3s) or `new_order.wav` (~3s). No new files needed unless owner wants a specific tone. |
| Other notification types | Non-YTC notifications (table attend, item ready, etc.) keep one-shot play behavior |

## 4. Technical Approach

### soundManager.js — new `playLoop` method
```javascript
playLoop(soundKey) {
  if (!this.isEnabled) return;
  if (soundKey === 'silent') { this.stop(); return; }
  const path = SOUND_FILES[soundKey];
  if (!path) return;
  this.stop();
  const audio = this.audioCache[soundKey]
    ? this.audioCache[soundKey].cloneNode()
    : new Audio(path);
  audio.loop = true;  // HTML5 native seamless loop
  this.currentAudio = audio;
  audio.play().catch((err) => {
    console.warn('[SoundManager] PlayLoop blocked:', err.message);
    this.currentAudio = null;
  });
}
```

### NotificationContext.jsx — conditional loop
```javascript
if (isConfirmOrderNotification(data, resolvedSound)) {
  soundManager.playLoop(resolvedSound);  // loop until action
} else {
  soundManager.play(resolvedSound);      // one-shot
}
```

### Existing stop triggers (already wired)
- Snooze → `soundManager.stop()` ✓
- Silent notification → `soundManager.stop()` ✓
- New notification → `soundManager.play/playLoop` stops previous ✓
- Accept/Reject → add `soundManager.stop()` before action

## 6. QA Checklist

1. Web order arrives → short tone loops continuously
2. Click Accept → sound stops immediately
3. Click Reject → sound stops immediately
4. Click Snooze/Mute → sound stops immediately
5. Silent notification → sound stops
6. Second web order while first is ringing → new loop replaces old
7. Non-YTC notifications → still play once (no loop)
8. Sidebar Silent Mode off → no sound at all
9. Page refresh → sound stops (no persistence)
