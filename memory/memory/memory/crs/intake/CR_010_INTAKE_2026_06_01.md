## 1. Requirement (owner-reported)

Today POS treats every menu item as discrete unit-based (1 burger, 1 packet of sweets). Going forward, some items will be sold by weight (e.g., sweets sold by kilogram). The Product API will provide two new fields per item: item_unit (e.g., "Kg") and item_unit_price (e.g., "10"). POS must consume these fields across the full order flow: item selection, quantity/weight capture, edit-order, line-item & bill calculation, and final order payload submission. Impact analysis must cover: menu/product API mapping, cart logic, edit-order flow, bill calculation, and order payload compatibility. Existing piece-based items must remain unchanged (backwards-compatible).

## 2. Expected Behavior

Menu/Product API mapping reads item_unit + item_unit_price and exposes them on cart lines. Item selection UX adapts when item_unit != piece/unit (shows Weight input instead of integer qty stepper). Cart logic computes line_total = weight x item_unit_price for weight items; piece-based items unchanged. Edit-order flow preserves and allows re-entry of captured weight. Bill calculation (subtotal, taxes, discounts, round-off, grand total) and Print/KOT templates handle weight items without regression. Final order payload to backend carries weight + unit + unit_price fields (contract TBD in impact analysis). Backwards-compatible with existing piece-based items.
